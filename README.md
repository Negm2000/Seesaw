# Seesaw Project File Guide

Welcome to the Seesaw project. This file serves as a guide to help you understand the purpose of each script and folder in the repository so you know exactly where to look!

## 🚀 Getting Started

**`startup.m`**
Run this script *every time* you open MATLAB. It adds all the necessary folders (and their subfolders) to your MATLAB path so that scripts can find each other and your Simulink models can locate parameter variables.

---

## 📂 The `scripts/` Directory
This is where the actual logic lives. It's broken down into modules based on the project phase:

### 1. `config/` (The Core Parameters)
* **`seesaw_params.m`**: The most important script in the codebase. It holds all the hardware constants (masses, motor torque, saturation, lengths). It also pre-calculates the linearised **State-Space matrices** (`A`, `B`, `C`, `D`) for both the single cart and the full seesaw. Almost every Simulink model and script depends on this.

### 2. `modeling/` (Physics and Simulation)
* **`modeling_pipeline.m`**: The master script for **System Identification**. You run this to compare your analytical mathematical model to the actual hardware. It automatically adjusts friction parameters (`B_eq`) to match the hardware data.
* **`cart_on_table_model.m`**: A nonlinear simulation for Phase 1 (just the cart on a flat track). Used to verify cart dynamics alone.
* **`seesaw_nonlinear_model.m`**: A nonlinear simulation for the full cart+seesaw coupled system. It validates how much error your linearised state-space model introduces.

### 3. `setup/` (Model Building)
* **`build_simulink_models.m`**: Automatically assembles the Simulink models (`.slx` files) from MATLAB code, wiring up the State-Space blocks and the QUARC hardware blocks if you are connected to the lab bench. 
* **`frequency_setup.m`**: Builds the specific Simulink model designed to run a "chirp" frequency sweep on the hardware to collect raw performance data.

### 4. `analysis/` (Post-Hardware Verification)
* **`frequency_analysis.m`**: Takes the raw hardware frequency sweep data, compares it against your math model's Bode plots, and automatically runs an optimizer (auto-tuner) to find the exact unmodeled friction values.
* **`validate_against_hardware.m`**: A quick tool to overlay simulation time-domain graphs with actual hardware time-domain graphs so you can visually verify "Did my simulation match reality?".

### 5. `control/` (Controller Design)
* **`control_pipeline.m`**: The script where you design the active feedback loop (like LQR) to actually balance the seesaw. It calculates the `K` feedback matrix and simulates the closed-loop performance.

---

## 📂 Other Important Directories

* **`models/`**: Holds your Simulink `.slx` files.
  * *`IP02_CartOnTable.slx`*: Phase 1: Cart-only model.
  * *`Seesaw_Full.slx`*: Phase 2: Full balancing model.
* **`data/`**: 
  * *`data.mat`*: The default file where QUARC hardware logs its sensor recordings.
  * *`tuned_params.mat`*: Generated by your analysis scripts. This file stores your fine-tuned friction parameters so your controller can use the most accurate model.
* **`docs/`**: Reading materials! Contains the official hardware `manuals/`, your presentation `reports/`, and academic `references/`.
* **`src/`**: Contains custom S-Function blocks (`cart_table_sfun.m`, `seesaw_plant_sfun.m`) if you prefer defining plant dynamics directly in code instead of standard State-Space blocks.
* **`slprj/`**: Autogenerated C-code cache created by Simulink and QUARC. You can safely ignore this folder.
